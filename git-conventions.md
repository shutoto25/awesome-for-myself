# Git運用規約

Git操作（コミット、プッシュ、PR作成、ブランチ操作）を行う際は、以下のルールに従うこと。

## 原則

日本語で記載すること。

## コミットメッセージ

Conventional Commits仕様に従う。

### 形式

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### Type

| Type | 説明 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響しない変更（空白、フォーマット等） |
| `refactor` | バグ修正でも機能追加でもないコード変更 |
| `perf` | パフォーマンス改善 |
| `test` | テストの追加・修正 |
| `build` | ビルドシステムや外部依存関係の変更 |
| `ci` | CI設定の変更 |
| `chore` | その他（ソースやテストを変更しない） |
| `revert` | 以前のコミットを取り消す |

### Scope

変更対象のモジュール名、機能名、レイヤー名を括弧内に記述する。
プロジェクト構造から適切なscopeを判断すること。

### ルール

- descriptionは簡潔に（50文字以内推奨）
- bodyには変更の理由や背景を記載
- 複数の異なるTypeの変更は別々のコミットに分ける
- 破壊的変更がある場合はフッターに `BREAKING CHANGE:` を記載するか、type/scopeの後に `!` を付ける

## ブランチ運用

- 明示的な指示がなければ現在のブランチでコミット・プッシュしてよい
- ブランチを切ってPR経由にしてほしい場合はユーザーが指示する
- ユーザーの指示なしにコミットやプッシュを勝手に実行しない（必ずユーザーの依頼を受けてから行う）

## Pull Request

### PRタイトル

コミットメッセージと同じConventional Commits形式。70文字以内。

```
<type>(<scope>): <description>
```

### PRボディ

`.github/PULL_REQUEST_TEMPLATE.md` のテンプレートに従う。

### PRの作成手順

1. 未コミットの変更がないか確認（あればコミットを提案）
2. 変更内容を確認してPRタイトル・ボディを作成
3. リモートにプッシュ
4. `gh pr create` でPRを作成
